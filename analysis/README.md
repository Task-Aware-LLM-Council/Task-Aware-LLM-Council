# Metrics Analysis for Task-Aware LLM Council

This folder contains **offline analysis scripts** used to compute system-level metrics for our project, on top of the outputs generated by the existing benchmarking pipeline (`benchmark_runner` + `benchmarking_pipeline`).[file:289]

The goal is to **not** touch the core evaluation code, but to read its JSONL outputs and produce aggregate metrics that we report in the final paper.

---

## Folder structure and dependencies

- `analysis/metrics_analysis.py`  
  Standalone script for computing:
  - Average tokens per question
  - Average number of model calls per question  
  (Later) Routing accuracy and latency summaries.

- Core pipeline outputs used:
  - Prediction JSONL files written by `benchmarking_pipeline.storage.append_prediction` under:
    - `results/benchmark_suite/<suite_name>/predictions/<run_id>/predictions/<dataset>__<model>.jsonl`[file:289]
  - (Later) Oracle and router decision logs will be simple JSONL/JSON files created by separate scripts.

**Important:** All analysis is **read-only** on `results/`; it never calls models or changes existing outputs.

## Metrics definitions

### 1. Average tokens per question

For a given policy (P1, P2, P3, P4), and a set of \(N\) questions:

- Let \(k_i\) be the number of model calls made for question \(i\).
- Each call \(j\) has:
  - `usage.input_tokens`
  - `usage.output_tokens`
  - (optionally) `usage.total_tokens` if the provider sets it.

Total tokens for question \(i\):

\[
T_i = \sum_{j=1}^{k_i} (\text{prompt\_tokens}_{ij} + \text{completion\_tokens}_{ij})
\]

Average tokens per question:

\[
\text{AvgTokensPerQuestion} = \frac{1}{N} \sum_{i=1}^{N} T_i
\]

Lower values indicate **lower computational cost** and usually lower latency for a fixed deployment.[web:200][web:301]

### 2. Average number of model calls per question

Using the same notation, for each question \(i\) we count how many prediction records \(k_i\) exist (successful calls with status `"success"`).

Average calls per question:

\[
\text{AvgCallsPerQuestion} = \frac{1}{N} \sum_{i=1}^{N} k_i
\]

- P1 (single model) should be ~1 call per question.
- P2 (flat council) will be ~number_of_models_per_council.
- P3/P4 should be between these extremes if routing is effective.

### 3. Routing accuracy (planned)

Routing accuracy measures how often a router chooses the same model(s) as an **oracle** that knows, for each question, which model performed best in P1.[web:301][web:300]

For each example \(i\):

- Oracle set \(\mathcal{O}_i\): models with the best score for that example in P1.
- Router choice \(\mathcal{R}_i\): set of models chosen by P3/P4.

We define a “hit” if the router’s choice intersects the oracle set:

\[
\text{hit}_i =
\begin{cases}
1 & \text{if } \mathcal{R}_i \cap \mathcal{O}_i \neq \emptyset \\
0 & \text{otherwise}
\end{cases}
\]

Routing accuracy:

\[
\text{RoutingAccuracy} = \frac{1}{N} \sum_{i=1}^{N} \text{hit}_i
\]

This isolates router quality independent of raw model quality.

Routing accuracy will be computed later once:
- P1 per-example scores are available (to build \(\mathcal{O}_i\)).
- P3/P4 write per-example router decision logs (`example_id`, `chosen_models`).

## How `metrics_analysis.py` works

### Inputs

The script takes a **suite directory** produced by `benchmark_runner`:

- Example: `results/benchmark_suite/DeepSeek-Coder-V2-Lite-Instruct`

Inside this directory we expect:

- `predictions/<run_id>/predictions/<dataset>__<model>.jsonl`

Each JSONL line is a `BenchmarkPrediction` record with keys like:

```json
{
  "dataset_name": "fever",
  "example_id": "776c60e14509da84a9c1a4527d53e208",
  "model": "deepseek-ai/DeepSeek-Coder-V2-Lite-Instruct",
  "status": "success",
  "latency_ms": 567.24,
  "usage": {
    "input_tokens": 160,
    "output_tokens": 3,
    "total_tokens": 163
  },
  ...
}
```

### Processing steps

1. **Discover prediction files**

   ```python
   # Finds suite_dir/predictions/*/predictions/*.jsonl
   pred_files = list(suite_dir.glob("predictions/*/predictions/*.jsonl"))
   ```

2. **Load and group by example**

   ```python
   records = load_predictions(path)
   grouped = group_by_example(records)  # example_id -> list of calls
   ```

3. **Compute averages**

   - For each `example_id`, sum `usage.total_tokens` (or `input+output`) over all calls → \(T_i\).
   - Average \(T_i\) over all examples in that file → `avg_tokens_per_question`.
   - Count number of calls per example → \(k_i\), average over examples → `avg_calls_per_question`.

4. **Aggregate across files**

   - The script prints per-file metrics (per dataset/model), then computes simple overall means across files as a quick global summary.

In the future, the same pattern will be extended to:

- Compute routing accuracy using oracle and router logs.
- Compute latency distributions (mean, median, p95) using `latency_ms`.

## How to run the analysis

From the project root, with your Python environment active:

```bash
python -m analysis.metrics_analysis \
  --suite-dir results/benchmark_suite/DeepSeek-Coder-V2-Lite-Instruct
```

Typical output (example):

```text
Found 5 prediction files under results/benchmark_suite/DeepSeek-Coder-V2-Lite-Instruct

results/.../predictions/fever__deepseek-ai_DeepSeek-Coder-V2-Lite-Instruct.jsonl:
  examples: 500
  avg_tokens_per_question: 132.47
  avg_calls_per_question: 1.00

...

=== Overall (simple mean across files) ===
avg_tokens_per_question: 145.32
avg_calls_per_question: 1.00
```

You can run this:

- **Locally**, on copied `results/` directories.
- **On CARC**, once P1/P2/P3/P4 runs are finished.
